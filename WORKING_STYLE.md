# Kai Working Style

Use this project with a senior-engineer, autopilot mindset.

## Default mode

- Inspect first, then implement.
- Make reasonable assumptions and keep going.
- Only stop when truly blocked.
- Finish the next concrete milestone end-to-end before summarizing.
- Prefer useful results over long explanations.

## Execution style

- Work in tight loops: inspect, change, validate, continue.
- Keep diffs focused and intentional.
- Prefer reusable workflows over one-off hacks.
- Do not leave generated assets or scripts disconnected from the project.
- If a path is clearly wrong, pivot quickly and preserve only what is reusable.

## Tooling priorities

- Prefer local-first tools: Ollama, Blender, Godot, Python, file-based memory.
- Assume an 8 GB RAM machine and avoid heavy always-on stacks.
- For visuals, prefer real references and maintainable Blender workflows over procedural guesswork.
- For code and assets, organize source, generated, and preview outputs into named folders.

## Validation rules

- Validate every meaningful change with the right command or runtime check.
- If a tool path is missing or broken, find the installed equivalent and continue.
- If something fails, debug it and try the next likely fix without waiting.

## Communication style

- Keep progress updates short and practical.
- Do not ask for confirmation between obvious next steps.
- Summaries should come after a real milestone, not before.

## Kai-specific priorities

- Prioritize making Kai usable and lifelike over side experiments.
- Prefer maintainable companion systems over temporary demo behavior.
- For realism, use actual Kai references, proper texturing workflows, and reusable animation/state systems.
